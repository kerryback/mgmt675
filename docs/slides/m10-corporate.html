Data Security

The Agent Evaluation Loop

An AI agent that queries your database does not just send one message — it runs a multi-turn loop.

1

Prompt

System prompt + user question sent to LLM

2

Generate

LLM writes SQL or Python

3

Execute

Code runs locally against your database

4

Evaluate

Results sent back to LLM for review

Steps 1–3 can be made safe. Step 4 is where real data may leak to the cloud provider — query results, error messages containing data values, or raw rows the LLM needs to interpret.

Protecting Step 1: Schema + Synthetic Rows

The system prompt should describe your database using schema and synthetic rows — never real data.

What to include

  • Table names, column names, data types
  • ~3 synthetic rows per table (fabricated, matching real formats)
  • Column descriptions and domain conventions
  • Databricks: ~3 synthetic rows raises SQL accuracy by ~6 pp

Why synthetic rows help

  • Reveal data formats (is status 'Active' or 1?)
  • Disambiguate columns (is code a country code or product code?)
  • Help the LLM write correct WHERE clauses
  • No real records are transmitted

Schema metadata is not PII — safe to share. The LLM writes the query; your server runs it.

Protecting Step 4: The Hard Part

When the agent loop sends results back to the LLM, real data leaves your network. Mitigations exist but each has gaps.

Mitigations

  • Aggregate only: send row counts and summary statistics, never raw rows
  • Error messages only: if SQL fails, send the error string (schema info, not data) for correction; if it succeeds, don’t send results back
  • Redaction layer: middleware that strips values matching PII patterns before sending to the LLM

Why Mitigations Are Not Foolproof

  • A query like “show the account with the highest balance” returns one row — that is PII
  • SQL errors can echo offending data values
  • Redaction is pattern-based and misses context-dependent sensitivity
  • “Don’t send results back” limits the agent to single-turn — you lose the error-correction loop that makes agents useful

Data Policies: Training and Retention

Before sending any data to a cloud AI provider, know the answers to two questions.

Will they train on your data?

  • Consumer tiers (free ChatGPT, free Claude) may use your input to improve models
  • API and enterprise tiers typically do not
  • Check your agreement and settings

How long do they keep it?

  • Providers retain prompts for a window (often 30 days) for safety monitoring
  • Enterprise contracts may offer zero retention
  • Know the policy before you send sensitive data

Even with zero-training guarantees, data still transits an external server. For regulated data, transit alone may violate compliance.

The Hybrid Pattern

For most organizations, the answer is not “all cloud” or “all on-prem” — it is hybrid.

Cloud AI (General Work)

  • Research, drafting, coding assistance
  • Non-sensitive data analysis
  • Frontier model quality
  • No infrastructure to manage

On-Prem AI (Sensitive Data)

  • Client data under NDA
  • HIPAA, GDPR, SOX-covered records
  • Material non-public financial information
  • Defense and classified environments

The schema-only pattern lets you use cloud AI for the thinking while keeping sensitive data on-prem for the execution. But when even the evaluation loop must stay local, you need on-prem models.

Private Cloud: On-Prem Managed by Experts

Private cloud means AI infrastructure on your premises, managed by an outside provider. You get data control without building the IT team yourself.

How it works

  • Hardware sits in your data center (or a dedicated facility)
  • A vendor (HPE, Dell, IBM) manages the infrastructure
  • Your data never leaves your network
  • The vendor handles updates, scaling, and maintenance

Enterprise examples

  • DISA (U.S. DoD): $931M HPE private cloud
  • Barclays: 50K+ workloads to private cloud
  • 2degrees (NZ): sovereign AI for data residency

Antonio Neri, CEO of HPE: “In the AI era, where data sovereignty, privacy and control of data matter, the industry is seeing a shift back to private clouds.”

If mitigations for the evaluation loop were foolproof, this market wouldn’t exist. Private cloud exists because some data cannot risk transit through any external server, even transiently.

What On-Prem AI Actually Requires

More Than a Model

Choosing on-prem means building an entire AI platform, not just downloading a model.

The Model Layer

  • Open-source LLM (Llama, Qwen, Mistral)
  • GPU servers to run inference
  • Model serving software (Ollama, vLLM)

The API Layer

  • Internal API endpoint (like OpenAI’s, but on your network)
  • Authentication, rate limiting, logging
  • Lets any internal app call the model programmatically

The Interface Layer

  • Chat UI for non-technical users
  • System prompt management
  • Conversation history and audit logs

With a cloud provider, you get all three out of the box. On-prem, you build each one.

The Developer Tooling Gap

For AI-assisted coding, the gap is even wider. A coding agent is not just an LLM — it’s a harness of tools wrapped around the model.

What a Coding Agent Needs

  • File read/write with permission controls
  • Terminal execution in a sandbox
  • Git integration (commits, diffs, branches)
  • Context management (which files to feed the model)
  • Multi-agent orchestration
  • Memory and session persistence

What Cloud Providers Bundle

  • Anthropic: Claude Code (~500K lines of tooling)
  • GitHub: Copilot (IDE-integrated agent)
  • Google: Gemini Code Assist
  • All include the harness for free with API access
  • Continuously updated by large engineering teams

The model is the brain. The harness is everything else. On-prem means building the harness yourself.

Case Study: The Claude Code Leak

On March 31, 2026, Anthropic accidentally published Claude Code’s full source code — 512,000 lines of TypeScript — via an npm packaging error.

What it revealed

  • ~40 permission-gated tools (file, terminal, git, web, IDE)
  • Three-layer memory architecture
  • Multi-agent orchestration via prompts
  • Granular sandboxing and access control
  • Background “dream” mode for autonomous work

What happened next

  • claw-code: open-source reimplementation hit 160K GitHub stars in days
  • Provider-agnostic: works with Claude, OpenAI, or local models via Ollama
  • Other alternatives: Aider, OpenHands, Continue.dev
  • All are less mature than the cloud originals

The leak made the cost of developer tooling visible. 500K lines of code is what sits between a raw model and a productive developer.

The Build-vs-Buy Reality

The TCO of on-prem AI is not just GPU costs.

Cloud (e.g., Anthropic API)

  • Model + API + coding agent included
  • Pay per token (~$3–15 per million)
  • No infrastructure team needed
  • Continuous model improvements
  • Risk: data leaves your network

On-Prem

  • GPU hardware ($50K–$500K+)
  • Model serving and API infrastructure
  • Build or assemble developer tooling
  • Ongoing maintenance and model updates
  • Engineering team to support all of the above
  • Benefit: complete data control

Implement Your Own On-Prem

Ollama: Local LLMs in One Installer

Ollama lets you run open-source models on your laptop — no cloud, no API key, no data leaving your machine.

How it works

  1. Download and run the Ollama installer
  2. Pull a model (e.g., llama3.2)
  3. Chat locally via a browser UI or the terminal
  4. All inference runs on your CPU/GPU

Available models

  • Llama 3.1 (Meta, 8B) — strong reasoning
  • Llama 3.2 (Meta, 3B) — faster, lighter
  • Mistral, Gemma, Phi — many options
  • Models download once and run offline

Exercise: Running a Local Model

Setup

  1. Go to ollama.com and download the installer for your OS

  2. Run the installer (no terminal needed)

  3. Once installed, open a terminal and run:

    ollama run llama3.2

  4. Then try:

    ollama run llama3.1

The prompt

Give each model this prompt and compare their responses:

“A private equity firm is evaluating a manufacturing company with $50M in revenue, 12% EBITDA margins, and 8% annual revenue growth. Comparable transactions have been done at 7–9x EBITDA. What is a reasonable valuation range, and what are the top three risks you would flag before proceeding?”

What to notice: quality of reasoning, how each model handles uncertainty, and whether either refuses or hedges differently.

Both models run entirely on your machine. No data is sent anywhere.

The Traditional Enterprise Stack

How Companies Run Today

A typical mid-market company ($500M revenue) runs 10+ enterprise systems across divisions.

CRM

  • Salesforce, HubSpot, or legacy
  • Pipeline, contacts, deals
  • Each division may use a different one

Finance / ERP

  • NetSuite, SAP, QuickBooks
  • GL, invoices, budgets
  • Often different systems per division

Operations & HR

  • SAP, Oracle SCM, Workday
  • Supply chain, inventory, headcount
  • Typically centralized

These systems don’t talk to each other. Getting a cross-system answer requires ETL pipelines, data warehouses, or analyst time.

The Cross-System Problem

A CEO asks: “Which customers buy from multiple divisions?”

1

Request

CEO emails the BI team

2

Gather

Analyst queries 3 CRMs separately

3

Reconcile

Match “GE” / “GE Industrial” / “GE Safety”

4

Deliver

3 weeks later: a spreadsheet

The traditional answer is a data warehouse ($500K–$5M, 6–18 months to build). The AI alternative: query each system where it lives and merge on the fly.

Realistic Data Friction

The simulation includes the same data quality problems real enterprises face.

Naming Conflicts

  • Salesforce: CamelCase columns
  • Legacy CRM: snake_case columns
  • Same customer, different names across divisions

Format Mismatches

  • Legacy CRM stores dates as text: "03/15/2025"
  • Other systems use proper date types
  • Revenue in different currencies or fiscal calendars

No Cross-System Joins

  • The agent cannot join across systems in SQL
  • Must query each system separately
  • Merges data with Python + fuzzy matching

This is exactly how a real enterprise AI tool would need to work.

The AI Agent Architecture

The agent writes SQL for each system, executes it, then uses Python to merge, reconcile, and visualize the combined results.

The Counter-Argument

Why ERP Isn’t Dead

The SaaSpocalypse thesis has real limits. Enterprise software does things AI agents cannot easily replace.

What ERP Does Beyond Queries

  • Transaction processing (orders, invoices, payments)
  • Workflow enforcement (approval chains, compliance gates)
  • Audit trails with legal standing
  • Role-based access control at the field level
  • Real-time integrations (inventory, shipping, payments)

What AI Agents Can’t Do (Yet)

  • Process a purchase order end-to-end
  • Enforce SOX compliance controls
  • Guarantee data consistency across systems
  • Replace the system of record
  • Provide legally defensible audit trails

ERP is a system of record. AI agents are a system of insight. These are different jobs.

The Integration Layer Is the Real Target

The $2T thesis is partially right, partially wrong.

What AI will displace

  • BI dashboards and ad-hoc reporting
  • Data integration middleware (for read-only queries)
  • Analyst time spent on data wrangling
  • Tier 1 support triage
  • Manual report generation

What survives

  • Transaction processing engines
  • Compliance and audit infrastructure
  • Workflow orchestration with legal standing
  • Master data management
  • Systems of record

The disruption isn’t “AI replaces SAP.” It’s “AI replaces the $200K/year you spend making SAP’s data accessible.”

What This Means for Finance Professionals

Near Term (2026–2027)

  • AI agents supplement BI tools, not replace them
  • Cross-system queries become self-service
  • Reporting cycles compress from weeks to hours
  • New role: AI verifier — checking agent output, not producing it

Medium Term (2027–2029)

  • ERP vendors embed AI agents natively (SAP Joule, Salesforce Agentforce)
  • Standalone BI tools lose market share
  • Data warehouse projects face “build vs. agent” decisions
  • Compliance frameworks catch up

Long Term (2029+)

  • The “reporting layer” is fully AI-native
  • ERP becomes invisible infrastructure (like databases today)
  • Competitive advantage shifts from data access to data judgment
  • The CFO’s question isn’t “can we get the data?” — it’s “can we trust the answer?”

The Verification Problem at Enterprise Scale

Everything from Module 7 applies here (and Module 13 for details). An AI agent querying 10 systems makes all the same errors — at enterprise scale.

The demo looked impressive. But…

  • Did the fuzzy matching miss a customer?
  • Did the date format mismatch skew Q4 numbers?
  • Did the agent silently drop rows with NULL values?
  • Did it average across won and lost deals?

The verification checklist still applies

  1. Sanity-check with known answers
  2. “Show me the SQL” for each system
  3. Cross-check by rephrasing
  4. Spot-check edge cases and merge logic

The more systems involved, the more places for silent errors.

The SaaSpocalypse isn’t about whether AI can query your data.

It’s about whether anyone will trust the answer enough to act on it.

AI Adoption Roadmap

The Sandbox-Audit-Deploy Pattern

Before deploying any AI workflow to your team, follow three phases.

🧪

  1. Sandbox

Test with dummy data

🔍

  1. Audit

Review logs and logic

🚀

  1. Deploy

Roll out with monitoring

Test with synthetic data, review logs and logic, then roll out with weekly monitoring.

Adoption Timeline: From Quick Wins to Strategic

Each adoption stage maps to a trust level.

  1. Quick wins (weeks) — light verification:
    • Chat for research, summarization, drafts
    • Upload spreadsheets for ad-hoc analysis
  2. Medium-term (months) — test-before-deploy:
    • Skills for recurring workflows
    • MCP integrations and web apps
  3. Strategic (quarters) — full audit trail:
    • Production apps with authentication
    • AI usage policy and governance framework

Small Language Models and Fine-Tuning

Improving Reliability: Fine-Tuning and SLMs

A complementary approach to verification: make the model itself more reliable for your tasks.

Fine-tuning an LLM

  • Train on your domain data (10-Ks, memos, deal docs)
  • Model learns your terminology and conventions
  • Fewer silent errors on domain-specific tasks

Small Language Models (SLMs)

  • ~1–8B parameters (vs. hundreds of billions)
  • Fine-tuned in hours on a single GPU
  • Run on-premise — data never leaves your network
  • Specialized small models can outperform large models on narrow tasks

Why SLMs for Agentic Systems?

Belcak et al. (2025), “Small Language Models are the Future of Agentic AI”:

  • Most agent subtasks are repetitive, scoped, and non-conversational — classifying documents, extracting fields, formatting outputs
  • SLMs fine-tuned on a single task outperform LLMs on format compliance and consistency
  • 10–30x cheaper in latency, energy, and compute vs. large models
  • On-device deployment enables real-time, offline inference with full data privacy
  • Agents can use multiple models: an LLM for complex reasoning, SLMs for routine subtasks

From LLM to SLM: The Conversion Process

The paper proposes a six-step migration path:

📝

  1. Log

Agent interactions (prompts, responses, tool calls)

🧹

  1. Curate

Remove PII, filter for quality

🔗

  1. Cluster

Identify recurring task patterns

📐

  1. Select

SLM sized to each task’s complexity

⚙️

  1. Fine-tune

LoRA with 10K–100K examples

🔄

  1. Repeat

Retrain as usage patterns evolve
