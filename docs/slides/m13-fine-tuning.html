What is Fine-Tuning?

What is Fine-Tuning?

A pretrained LLM already knows language. Fine-tuning adjusts its weights on a smaller, task-specific dataset so it behaves differently:

  • Adopt a specific tone or format (e.g., always respond as a financial analyst)
  • Learn domain terminology and conventions
  • Follow company-specific rules and workflows

Analogy: a pretrained model is like hiring a smart generalist. Fine-tuning is specialized on-the-job training.

Fine-Tuning vs. Prompting vs. RAG

Prompting / Skills

  • Change behavior via instructions
  • No training required
  • Limited by context window
  • Easy to iterate

RAG

  • Inject external knowledge at query time
  • No model weights changed
  • Answers grounded in retrieved documents
  • Great for factual Q&A

Fine-Tuning

  • Change the model’s weights
  • Learns patterns, tone, and format
  • Knowledge baked into the model
  • Best for consistent, specialized behavior

When to Fine-Tune

Fine-tuning is worth the effort when:

  • You need consistent behavior across thousands of queries (not one-off tasks)
  • Prompting alone can’t capture the nuance (complex formatting, domain jargon, regulatory style)
  • You want lower latency and cost than a large model with a long system prompt
  • You have (or can generate) high-quality training examples

Fine-tuning is not the right tool when:

  • Your data changes frequently (use RAG instead)
  • You need answers from specific documents (use RAG)
  • A well-crafted prompt already works (keep it simple)

The Fine-Tuning Process

Step 1: Collect Training Data

Training data is a set of input/output pairs — examples of what you want the model to do.

Input Output
“Summarize this 10-K risk factor…” “The company faces supply chain risk due to…”
“Classify this earnings call sentence…” “Positive guidance”
“Write a credit memo for…” “Credit Assessment: BBB+ …”

Start small. Google’s Gemma docs note that behavioral changes can appear with as few as 20 examples. Build from there.

Step 2: Choose an Approach

Full Fine-Tuning

  • Update all model parameters
  • Maximum flexibility
  • Requires significant GPU memory and compute
  • Produces a full copy of the model

LoRA (Parameter-Efficient)

  • Freeze the original weights
  • Train only small adapter matrices
  • 10,000x fewer trainable parameters
  • Same quality, fraction of the cost

In practice, almost everyone uses LoRA or a similar parameter-efficient method. Full fine-tuning is reserved for large-budget projects.

LoRA: The Key Idea

Why LoRA Works

The LoRA paper (Hu et al., 2021) made a key observation: when you fine-tune a large model, the weight changes have low intrinsic rank.

The changes needed to adapt a model to a new task live in a much lower-dimensional space than the full parameter space.

This means you don’t need to update billions of parameters — you can capture the same adaptation with far fewer.

LoRA: How It Works

Instead of learning a full weight update \(\Delta W\) (dimensions \(d \times d\)), decompose it into two small matrices:

\[\Delta W = A \times B\]

where \(A\) is \(d \times r\) and \(B\) is \(r \times d\), and \(r\) (the rank) is tiny: typically 4, 8, or 16.

Full Fine-Tuning

  • Update \(d^2\) parameters per layer
  • For \(d = 4096\): 16.8 million parameters
  • Full copy of model weights

LoRA (rank 16)

  • Update \(2 \times d \times r\) parameters per layer
  • For \(d = 4096\), \(r = 16\): 131,072 parameters
  • 128x reduction per layer

LoRA at Inference Time

During fine-tuning, the original weights are frozen:

\[y = W_{\text{original}}(x) + A \times B(x)\]

At deployment, merge the adapter into the base weights:

\[W_{\text{new}} = W_{\text{original}} + A \times B\]

  • No inference latency penalty — the merged model runs at normal speed
  • Adapter files are small (MBs, not GBs)
  • Swap adapters on the same base model for different tasks

QLoRA: Even More Efficient

QLoRA (Dettmers et al., 2023) adds quantization on top of LoRA:

  1. Compress base model weights to 4-bit precision (from 16-bit) — 4x memory savings
  2. Train LoRA adapters on top of the quantized model
  3. Paged optimizers handle memory spikes

Result: fine-tune a 65B-parameter model on a single 48GB GPU. Training is ~39% slower due to quantization overhead, but quality is comparable to full-precision LoRA.

Hands-On: Fine-Tuning Gemma

What We’ll Do

Fine-tune Google Gemma 3 1B to classify financial news sentiment using QLoRA.

Setup

  • Google Colab (free T4 GPU)
  • Hugging Face account (accept Gemma license)
  • ~15 minutes to train

Stack

  • transformers — load and run the model
  • peft — apply LoRA adapters
  • trl — supervised fine-tuning trainer
  • bitsandbytes — 4-bit quantization (QLoRA)

Step 1: Install and Login

from huggingface_hub import login

You need to accept Google’s Gemma license at huggingface.co/google/gemma-3-1b-it before downloading.

Step 2: Load the Model with QLoRA

from transformers import AutoTokenizer, AutoModelForCausalLM, BitsAndBytesConfig
from peft import LoraConfig, get_peft_model
import torch

# 4-bit quantization config (QLoRA)
bnb_config = BitsAndBytesConfig(
    load_in_4bit=True,
    bnb_4bit_quant_type="nf4",
    bnb_4bit_compute_dtype=torch.bfloat16,
)

model_id = "google/gemma-3-1b-it"
tokenizer = AutoTokenizer.from_pretrained(model_id)
model = AutoModelForCausalLM.from_pretrained(
    model_id, quantization_config=bnb_config, device_map="auto"
)

The 1B model loads in ~800 MB with 4-bit quantization (vs. ~2 GB at full precision).

Step 3: Attach LoRA Adapters

lora_config = LoraConfig(
    r=16,                          # rank — higher = more capacity
    lora_alpha=32,                 # scaling factor
    target_modules="all-linear",   # apply to all linear layers
    lora_dropout=0.05,
    task_type="CAUSAL_LM",
)

model = get_peft_model(model, lora_config)
model.print_trainable_parameters()
# trainable params: 3,407,872 || all params: 999,698,432 || trainable%: 0.34%

Only 0.34% of parameters are trainable — the rest are frozen.

Step 4: Prepare Training Data

from datasets import Dataset

# Financial sentiment classification examples
examples = [
    {"text": "Revenue surged 23% year over year, beating analyst estimates.",
     "label": "positive"},
    {"text": "The company announced 2,000 layoffs amid declining margins.",
     "label": "negative"},
    {"text": "Q3 results were in line with expectations, no change to guidance.",
     "label": "neutral"},
    # ... add 50-200 more examples
]

def format_example(example):
    prompt = f"""Classify the financial sentiment of this text.
Text: {example['text']}
Sentiment:"""
    return {"text": f"{prompt} {example['label']}"}

dataset = Dataset.from_list(examples).map(format_example)

Step 5: Train

from trl import SFTConfig, SFTTrainer

training_args = SFTConfig(
    output_dir="./gemma-finance-sentiment",
    num_train_epochs=3,
    per_device_train_batch_size=4,
    learning_rate=2e-4,
    max_seq_length=256,
    logging_steps=10,
)

trainer = SFTTrainer(
    model=model,
    args=training_args,
    train_dataset=dataset,
)

trainer.train()

On a Colab T4 GPU, this takes ~10–15 minutes for 200 examples × 3 epochs.

Step 6: Test the Result

# Test on a new example the model hasn't seen
prompt = """Classify the financial sentiment of this text.
Text: Net income fell 12% as rising input costs squeezed margins.
Sentiment:"""

inputs = tokenizer(prompt, return_tensors="pt").to(model.device)
outputs = model.generate(**inputs, max_new_tokens=5)
print(tokenizer.decode(outputs[0], skip_special_tokens=True))
# → "... Sentiment: negative"

Step 7: Save and Share

# Save the LoRA adapter (just a few MB)
model.save_pretrained("./gemma-finance-sentiment")

# Later, reload on any machine:
from peft import PeftModel
base = AutoModelForCausalLM.from_pretrained(model_id, ...)
model = PeftModel.from_pretrained(base, "./gemma-finance-sentiment")

# Or merge into a single model file:
merged = model.merge_and_unload()
merged.save_pretrained("./gemma-finance-merged")

The adapter is ~14 MB. The full merged model is ~2 GB. Both produce identical results.

Distillation: LLM → Small Model

Why Not Just Use the Big Model?

A powerful pattern: use a large model to generate training data, then fine-tune a smaller model on it.

Why bother? Why not just use the LLM directly?

Cost

Large models cost 10–30x more per query than small models

Generate 50K examples once ($200–500), then run the small model forever at a fraction of the cost

Speed

A 7B model generates responses in 50–200ms on modest hardware

Large API models take 1–5 seconds

Critical for real-time applications

Privacy

A fine-tuned small model runs entirely on your infrastructure

No data leaves your network

Essential for healthcare, legal, and financial applications (HIPAA, GDPR)

Distillation in Practice

Well-known examples of LLM → small model distillation:

  • Stanford Alpaca: GPT-3.5 → LLaMA 7B. Cost: $600. Comparable performance on many tasks.
  • Microsoft Orca: GPT-4’s detailed reasoning traces (not just answers) → smaller model. Capturing the teacher’s thinking process improves quality.
  • Microsoft Phi: LLM-generated “textbook quality” synthetic data → 1.3B–3B parameter models that punch far above their weight.

The pattern works best for specific, repeatable tasks at scale: classify emails, extract fields, generate reports. For diverse, one-off tasks, just use the LLM directly.

Frameworks and Tools

Fine-Tuning Frameworks

Google’s Gemma documentation lists several popular tools:

Beginner-Friendly

  • Hugging Face + PEFT — most popular; LoRA built in
  • Unsloth — 2x faster training, lower memory
  • Keras — familiar API, LoRA support

Production / Scale

  • Axolotl — config-driven, many model architectures
  • LLaMA Factory — GUI for fine-tuning
  • Google Cloud Vertex AI — managed service, multi-GPU

For most use cases, start with Hugging Face + PEFT or Unsloth. Both support LoRA/QLoRA out of the box and have extensive documentation.

Step 3: Test Your Model

Google recommends three categories of test cases:

Success Tests

Inputs the model should always handle correctly

  • Core use case queries
  • Common variations
  • Expected formats

Failure Tests

Inputs the model should refuse or flag

  • Out-of-scope requests
  • Adversarial prompts
  • Requests for confidential data

Boundary Tests

Edge cases that test limits

  • Ambiguous inputs
  • Missing context
  • Mixed-language queries

Step 4: Deploy

For LoRA-tuned models, deployment is straightforward:

  • Deploy the base model + the LoRA adapter (a small file)
  • Or merge the adapter into the base weights for a single model file
  • Serve with standard inference tools (vLLM, TGI, Ollama)

Multiple LoRA adapters can share a single base model. A compliance adapter, a customer-service adapter, and a research adapter can all run on the same GPU, swapping adapters per request.
