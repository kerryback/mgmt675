MGMT 675: Generative AI for Finance

What Is an Agent?

The Claude API lets you build programmatic agents — applications that:

  1. Maintain a conversation with a user
  2. Have access to tools (functions they can call)
  3. Decide which tools to use based on what the user asks
  4. Chain multiple tool calls before responding

This is different from a script that always runs the same steps. The agent reasons about what information it needs and fetches it on the fly.

Our Example: Portfolio Review Agent

We’ll build a portfolio analyst that can:

  • Look up current holdings and tax lots
  • Query a live price database (MotherDuck)
  • Compute returns, correlations, and statistics in Python
  • Compare sector weights against targets
  • Identify tax-loss harvesting opportunities
  • Find the most correlated Strong Buy replacement

The agent chooses its own path. Asking “review the portfolio” triggers different tools than “find tax-loss candidates in energy.”

Why an Agent, Not a Script?

A Script

  • Fixed sequence of steps
  • Same analysis every time
  • Can’t handle follow-up questions
  • New analysis = new code

An Agent

  • Chooses tools based on context
  • Different path for each question
  • Conversational — remembers context
  • New analysis = just ask

The agent uses the same five tools to answer hundreds of different questions. You write the tools once; the agent figures out how to combine them.

Architecture

Five tools. Three return mock portfolio data. One executes SQL against a live price database. One runs Python for analytics.

Step 1: Define the Tools

Tool Definitions Are JSON Schema

Each tool has a name, description, and input_schema. The description tells the agent when and why to use the tool.

tools = [
    {
        "name": "get_holdings",
        "description": (
            "Retrieve all portfolio holdings with tax lot detail. "
            "Returns each position's ticker, sector, and individual "
            "lots (date acquired, shares, cost basis per share)."
        ),
        "input_schema": {
            "type": "object",
            "properties": {},
            "required": [],
        },
    },
    # ... more tools
]

Tools with no required inputs (like get_holdings) act as simple lookups. The agent calls them when it needs that data.

The SQL Tool

The most powerful tool lets the agent write its own queries:

{
    "name": "run_sql",
    "description": (
        "Execute a SQL query against the MotherDuck database.\n"
        "Available tables:\n"
        "  - sep: daily stock prices (ticker, date, close, closeadj, ...)\n"
        "  - tickers: stock metadata (ticker, name, sector, industry, ...)\n\n"
        "Use closeadj for return calculations (adjusted for splits/dividends)."
    ),
    "input_schema": {
        "type": "object",
        "properties": {
            "sql": {
                "type": "string",
                "description": "The SQL query to execute",
            }
        },
        "required": ["sql"],
    },
}

The agent composes SQL on the fly — correlations, drawdowns, returns — whatever the analysis requires.

The Recommendations Tool

{
    "name": "get_analyst_recommendations",
    "description": (
        "Get stocks rated 'Strong Buy' for a given sector. "
        "Use this to find replacement candidates when tax-loss "
        "harvesting creates a need to replenish sector exposure."
    ),
    "input_schema": {
        "type": "object",
        "properties": {
            "sector": {
                "type": "string",
                "description": "The sector (e.g., 'Technology')",
            }
        },
        "required": ["sector"],
    },
}

The agent only calls this after identifying a harvest opportunity — it doesn’t blindly fetch recommendations for every sector.

The Python Tool

The agent can compute analytics on data it retrieves:

{
    "name": "run_python",
    "description": (
        "Execute Python code and return printed output. "
        "pandas, numpy, and json are available. "
        "The namespace persists across calls."
    ),
    "input_schema": {
        "type": "object",
        "properties": {
            "code": {
                "type": "string",
                "description": "Python code to execute.",
            }
        },
        "required": ["code"],
    },
}

This enables multi-tool chains: run_sql fetches prices → run_python computes correlations. The agent decides what to compute based on the question.

Step 3: Tool Execution

Connecting to MotherDuck

import duckdb
from dotenv import load_dotenv
import os

load_dotenv("path/to/.env")

def run_sql_query(sql: str) -> str:
    """Execute SQL against MotherDuck, return JSON."""
    token = os.getenv("MOTHERDUCK_TOKEN")
    conn = duckdb.connect(f"md:ndl?motherduck_token={token}")
    try:
        df = conn.execute(sql).fetchdf()
        return df.to_json(orient="records", date_format="iso")
    except Exception as e:
        return json.dumps({"error": str(e)})
    finally:
        conn.close()

The agent writes SQL; this function executes it and returns a JSON string that the agent can interpret.

Executing Python Code

import io, sys

_python_namespace = {}  # persists across calls

def run_python_code(code: str) -> str:
    """Execute Python code, return printed output."""
    import numpy as np, pandas as pd
    _python_namespace.update({"pd": pd, "np": np, "json": json})

    old_stdout = sys.stdout
    sys.stdout = buf = io.StringIO()
    try:
        exec(code, _python_namespace)
        output = buf.getvalue()
        return output if output else "(no output)"
    except Exception as e:
        return f"Error: {e}"
    finally:
        sys.stdout = old_stdout

The shared _python_namespace lets the agent store a DataFrame in one call and use it in the next — e.g., fetch prices, then compute a correlation matrix.

The Tool Router

def execute_tool(name: str, inputs: dict) -> str:
    """Route a tool call to the appropriate handler."""
    if name == "get_holdings":
        return json.dumps(HOLDINGS)
    elif name == "get_target_allocation":
        return json.dumps(TARGET_ALLOCATION)
    elif name == "get_analyst_recommendations":
        sector = inputs["sector"]
        recs = STRONG_BUY.get(sector, [])
        return json.dumps({"sector": sector, "strong_buy": recs})
    elif name == "run_sql":
        return run_sql_query(inputs["sql"])
    elif name == "run_python":
        return run_python_code(inputs["code"])

This is your code — you control what each tool does. The agent only sees the tool’s name, description, and the result you return.

Step 4: The Agent Loop

The Core Pattern

import anthropic

client = anthropic.Anthropic()
messages = []

# User says something
messages.append({"role": "user", "content": user_input})

# Agent loop: keep going until it stops calling tools
while True:
    response = client.messages.create(
        model="claude-sonnet-4-6",
        system=SYSTEM_PROMPT,
        tools=tools,
        messages=messages,
        max_tokens=4096,
    )

    if response.stop_reason == "tool_use":
        # Agent wants to call tools — execute and feed results back
        ...
    else:
        # Agent is done — display its response
        break

The key insight: the loop keeps running until the agent has all the information it needs. It might call 1 tool or 5.

Handling Tool Calls

if response.stop_reason == "tool_use":
    # Save the assistant's message (contains tool_use blocks)
    messages.append({"role": "assistant", "content": response.content})

    # Execute each tool and collect results
    tool_results = []
    for block in response.content:
        if block.type == "tool_use":
            result = execute_tool(block.name, block.input)
            tool_results.append({
                "type": "tool_result",
                "tool_use_id": block.id,
                "content": result,
            })

    # Send results back to the agent
    messages.append({"role": "user", "content": tool_results})

The tool_use_id links each result to the corresponding call. The agent can make multiple tool calls in one turn.

The System Prompt

SYSTEM_PROMPT = """\
You are a portfolio analyst assistant. You help review a client
portfolio, identify tax-loss harvesting opportunities, and
recommend replacement securities.

When harvesting losses, sell the lots with the largest unrealized
loss first (highest cost basis relative to current price) using
specific lot identification for maximum tax benefit.

When recommending a replacement, compute return correlations
between the stock being sold and each Strong Buy candidate,
then recommend the most highly correlated candidate.

When writing SQL, use closeadj for return calculations, alias
tables (e.g., SELECT s.close FROM sep s), and cast date
(s.date::DATE) for comparisons.
"""

The system prompt gives the agent its strategy — domain expertise that guides tool selection. The full prompt also includes SQL and Python usage notes (see the complete script).

Step 5: The Conversation

Example Session

You: Review the portfolio — what’s the current sector allocation vs. target?

The agent calls get_holdings, then run_sql to get current prices, computes market values by sector, and compares to get_target_allocation.

You: Find tax-loss harvesting opportunities in Technology.

The agent calls run_sql to get current prices for tech holdings, compares against cost basis for each lot, and identifies the biggest losers (highest cost basis vs. current price).

Example Session (continued)

You: Harvest the INTC losses. What should I buy to replace the exposure?

The agent:

  1. Identifies INTC lots with the largest unrealized losses (sell those first)
  2. Calculates the realized loss
  3. Checks if selling moves Technology below target weight
  4. Calls get_analyst_recommendations("Technology") for Strong Buy candidates
  5. Calls run_sql to fetch daily prices for INTC and each candidate
  6. Calls run_python to compute return correlations
  7. Recommends the most highly correlated Strong Buy stock as the replacement

Seven tool calls across three tools, none predetermined. The path depends on what the human asks and what the data shows. Note: in practice, the replacement must not be “substantially identical” to the sold security, or the wash-sale rule disallows the loss. Choosing from a separate Strong Buy list (not ETFs tracking the same index) avoids this.

Deploying in a Corporate Environment

What Is Docker?

A Docker container packages your application, its dependencies, and its configuration into a single, portable unit.

Without Docker

  • “It works on my machine”
  • Install Python, pip, duckdb, anthropic…
  • Manage conflicting library versions
  • Secrets scattered across config files

With Docker

  • Identical environment everywhere
  • One command to build, one to run
  • Dependencies frozen in the image
  • Secrets injected at runtime

A container is like a lightweight virtual machine. It runs the same way on a developer’s laptop, a test server, or a cloud cluster.

Containerizing the Portfolio Agent

FROM python:3.12-slim

WORKDIR /app
COPY portfolio_agent.py .

RUN pip install anthropic duckdb python-dotenv pandas numpy

# Secrets are passed as environment variables at runtime,
# never baked into the image
# ANTHROPIC_API_KEY and MOTHERDUCK_TOKEN set at deploy time

CMD ["python", "portfolio_agent.py"]

The Dockerfile is a recipe. docker build creates the image; docker run starts the container. In production, companies deploy containers via orchestrators like Kubernetes or AWS ECS, which handle scaling, restarts, and secret management. The container also provides an isolation boundary for tools like run_python — code executes inside the container, not on the host.

Key Takeaways

  1. Tools are JSON Schema — define what the agent can do
  2. The system prompt — defines what the agent should do
  3. The agentic loop — keep feeding tool results until stop_reason != "tool_use"
  4. Human-in-the-loop — the user steers; the agent executes
  5. Chain tools — run_sql → run_python lets the agent fetch data then analyze it
  6. Mix real and mock data — live prices + curated portfolio = fast, compelling demo

About 200 lines of logic (plus data tables). The complexity lives in the agent’s reasoning, not in your application logic.
